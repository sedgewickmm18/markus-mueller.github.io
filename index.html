Why Do We Need Human Input?

Scenario: An AI assistant is asked to
"Complete the deployment review task"

Before acting, it needs to know:
• Action — approve or reject?
• Comment — a mandatory justification
• Confirmation — the action is irreversible once submitted

Without human input, the tool either guesses wrong or refuses to act.

MCP provides two patterns to solve this.

1. Explicit Prompting

User-Initiated Control

The client calls the tool without parameters; the server returns an error message explaining what is needed. The user provides the missing data and the client retries.

↓ See it in action

Example: Completing a Workflow Task

User says: "Complete the deployment review"

1st tool call (no parameters):

{
  "method": "tools/call",
  "params": {
    "name": "task_action",
    "arguments": {}
  }
}
Server returns an error:

{
  "isError": true,
  "content": [
    {
      "type": "text",
      "text": "Missing required parameters.
               Please provide 'action' (approve|reject)
               and 'comment' (mandatory justification)."
    }
  ]
}
Client shows form → User selects action & enters comment

✓ Works with any MCP client

Explicit Prompting — Retry with Parameters

2nd tool call (with user-provided data):

{
  "method": "tools/call",
  "params": {
    "name": "task_action",
    "arguments": {
      "action": "approve",
      "comment": "All tests passed, ready for production"
    }
  }
}
Server returns success:

{
  "content": [
    {
      "type": "text",
      "text": "Task completed: deployment review approved."
    }
  ]
}

✓ Zero ambiguity • ✓ User stays in control

Explicit Prompting — Key Points

  • Use cases: approving tasks, submitting reviews, completing workflow steps
  • Workflow: Call without params → Error with guidance → User provides data → Retry with params
  • Decision responsibility: lies with the client/user, not the model
  • Advantage: Universal compatibility — every MCP client can display an error
  • Limitation: Requires two round-trips; client must parse the error message

2. MCP Elicitation

Server-Initiated Pausing

The server pauses execution to request missing parameters via a JSON Schema form — no error, no retry needed.

↓ See it in action

Example: Completing a Workflow Task

User says: "Complete the deployment review"

Tool call (no parameters):

{
  "method": "tools/call",
  "params": {
    "name": "task_action",
    "arguments": {}
  }
}
Server pauses & sends elicitation request:

{
  "type": "elicitation",
  "title": "Complete Workflow Task",
  "message": "Please choose an action and provide a comment.",
  "requestedSchema": {
    "type": "object",
    "properties": {
      "action": {
        "type": "string",
        "enum": ["approve", "reject"],
        "description": "Select action"
      },
      "comment": {
        "type": "string",
        "description": "Mandatory justification"
      }
    },
    "required": ["action", "comment"]
  }
}
Client renders form → User selects action, enters comment & confirms

Elicitation — Key Points

  • Use cases: approval workflows, review submissions, task completion, confirmation dialogs
  • Workflow: Call without params → Server sends JSON Schema → Client renders form → User fills in → Single round-trip
  • Decision responsibility: Server manages state; user provides validation
  • Advantage: No retry needed — the server holds context and waits
  • Schema-driven: Supports text, boolean, enum, and nested objects

⚠️ Client Support

MCP Elicitation is a newer capability. Currently supported by:

  • Visual Studio Code (Copilot)
  • VS 2026
  • Claude Desktop
  • Claude Code
  • Cursor
  • MCPJam
  • Goose

Explicit Prompting works with all MCP clients by contrast.

Pros & Cons Comparison

Aspect Explicit Prompting MCP Elicitation
Trigger Client detects error & prompts user Server initiates when params are missing
UX Pattern Error message → form → retry Direct modal / dynamic form
Ambiguity Low — error message explains what's needed Low — schema constrains input
Client Compat. Works with every MCP client Requires elicitation-capable client
Round-trips Two calls (error + retry) Single call (server waits)
Flexibility Error message is free-form text Schema allows rich, validated input types
Implementation Simple — return error, then accept params More complex server-side state management
Safety User explicitly provides all data Server enforces schema validation

Key Takeaways

  • Start with Explicit Prompting — it's universal and simple
  • Add Elicitation when you want fewer round-trips and schema-validated forms
  • Both patterns complement each other — use the right tool for the job
  • Safety first: human-in-the-loop prevents costly mistakes in automated workflows
