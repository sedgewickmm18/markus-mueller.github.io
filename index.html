When tools need additional information from the user

Model Context Protocol — Architectural Patterns for User Intervention

Why Do We Need Human Input?

Scenario: An AI assistant is asked to
"Clean up old branches in the repository"

Before deleting anything, it needs to know:
• Which branches? — merged only, or stale too?
• How old is "old"? — 30 days? 90 days?
• Confirmation — some branches may still be in use

Without human input, the tool either guesses wrong or refuses to act.

MCP provides two patterns to solve this.

Two Approaches

1. Explicit Prompting

User-Initiated Control

The server returns broad results; the user decides how to narrow them down.

↓ See it in action

Example: Reviewing Pull Requests

User says: "Show me the open PRs"

Tool call:

{
  "method": "tools/call",
  "params": {
    "name": "list_pull_requests",
    "arguments": { "state": "open" }
  }
}
Server returns all open PRs:

{
  "content": [
    { "id": 101, "title": "Fix login bug", "author": "alice" },
    { "id": 102, "title": "Add dark mode",  "author": "bob"   },
    { "id": 103, "title": "Update README",  "author": "carol" }
  ]
}
Client renders list → User clicks PR #101

✓ Zero ambiguity • ✓ Works with any MCP client

Explicit Prompting — Key Points

  • Use cases: selecting files, picking branches, choosing team members
  • Workflow: Server lists all options → Client renders UI → User selects
  • Decision responsibility: lies with the client/user, not the model
  • Advantage: Universal compatibility — every MCP client can render a list
  • Limitation: Cannot prompt for data the server hasn't thought of in advance

2. MCP Elicitation

Server-Initiated Pausing

The server pauses execution to request missing parameters or user approvals via a JSON Schema form.

↓ See meaningful examples

Example 1: Destructive Action Confirmation

User says: "Delete the staging database"

Tool call:

{
  "method": "tools/call",
  "params": {
    "name": "delete_database",
    "arguments": { "environment": "staging" }
  }
}
Server pauses & sends elicitation request:

{
  "type": "elicitation",
  "title": "Confirm Database Deletion",
  "message": "This will permanently delete the staging database.",
  "requestedSchema": {
    "type": "object",
    "properties": {
      "confirm": {
        "type": "boolean",
        "description": "Type 'yes' to confirm"
      }
    },
    "required": ["confirm"]
  }
}
Client shows modal → User clicks "Confirm"

Example 2: Missing Required Parameter

User says: "Deploy to production"

Tool call:

{
  "method": "tools/call",
  "params": {
    "name": "deploy",
    "arguments": { "target": "production" }
  }
}
Server needs the branch name & asks via elicitation:

{
  "type": "elicitation",
  "title": "Deployment Details",
  "message": "Which branch should be deployed?",
  "requestedSchema": {
    "type": "object",
    "properties": {
      "branch": {
        "type": "string",
        "enum": ["main", "release/v2.1", "hotfix/auth-fix"],
        "description": "Select branch to deploy"
      },
      "rollback_on_failure": {
        "type": "boolean",
        "default": true,
        "description": "Auto-rollback on failure?"
      }
    },
    "required": ["branch"]
  }
}
Client renders form → User selects branch & confirms

Elicitation — Key Points

  • Use cases: confirming deletes, requesting branch names, entering API keys, approval workflows
  • Workflow: Server identifies missing data → Sends JSON Schema → Client renders form → User fills in
  • Decision responsibility: Server manages state; user provides validation
  • Advantage: Maintains context in complex multi-step workflows
  • Schema-driven: Supports text, boolean, enum, and nested objects

⚠️ Client Support

MCP Elicitation is a newer capability. Currently supported by:

  • Visual Studio Code (Copilot)
  • VS 2026
  • Claude Desktop
  • Claude Code
  • Cursor
  • MCPJam
  • Goose

Explicit Prompting works with all MCP clients by contrast.

Pros & Cons Comparison

Aspect Explicit Prompting MCP Elicitation
Trigger + User / Client initiates + Server initiates when needed
UX Pattern + List / Menu selection + Modal / Dynamic form
Ambiguity + Low — user sees all options + Low — schema constrains input
Client Compat. + Works with every MCP client – Requires elicitation-capable client
Multi-step Flows – User must drive each step + Server orchestrates automatically
Flexibility – Limited to pre-defined options + Schema allows rich input types
Implementation + Simple — just return data – More complex server-side state
Safety + User explicitly chooses + Server enforces confirmation

Key Takeaways

  • Start with Explicit Prompting — it's universal and simple
  • Add Elicitation when you need dynamic forms, confirmations, or multi-step workflows
  • Both patterns complement each other — use the right tool for the job
  • Safety first: human-in-the-loop prevents costly mistakes in automated tool chains
