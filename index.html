MCP Human in the Loop

Architectural Patterns for User Intervention

Implementation Alternatives

  1. Explicit Prompting (User-Initiated)
  2. MCP Elicitation (Server-Initiated)

1. Explicit Prompting

User-Initiated Control

The server returns broad results; the user decides how to narrow them down.

Workflow & Logic

  • Workflow: Server lists all options (files, PRs, etc.)
  • User Action: Client renders list; user clicks a selection
  • Decision: Responsibility lies with the client/user, not the model

Key Details

  • Use Case: Selecting a file, choosing a branch, or picking a team member
  • Advantage: Zero ambiguity. Works with any MCP client

2. MCP Elicitation

Server-Initiated Pausing

The server pauses execution to request missing parameters or approvals.

Workflow & Logic

  • Workflow: Server identifies missing data and sends a JSON schema request
  • User Action: Client renders a dynamic form or confirmation dialog
  • Decision: Server manages state; user provides validation

Key Details

  • Use Case: Confirming "delete," prompting for branch names, or requesting API keys
  • Advantage: Maintains context in complex, multi-step workflows

    participant U as User

    Note over U,S: Method 1: Explicit Prompting
    U->>C: "List PRs"
    C->>S: list_prs()
    S-->>C: Returns all PRs

    Note over U,S: Method 2: MCP Elicitation
    U->>C: "Merge PR #101"
    C->>S: merge_pr(id=101)
    S-->>C: Request Elicitation (Form)
    C->>U: Show Modal: "Confirm Merge?"
    U->>C: Approve
    C->>S: Submit Response
